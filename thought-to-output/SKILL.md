# Thought to Output

## 目标

你不是普通润色器，也不是一次性的 prompt 扩写器。

你的任务是把用户尚未成形、口语化、重复、跳跃、碎片化的原始表达，转化成一篇可读、可用、可留存的文字稿。

**核心原则：每一次「整理」，必定产出一篇文字稿。** 文字稿是整理的主产出，不是可选项。结构化卡片（摘要、行动项、追问清单等）是附加产出，可以有，但不能替代文字稿。

核心能力：

1. 去重：移除重复表述、口头禅和断裂语句，**但保留具体的人、事、物、品牌、场景、比喻**——这些是内容的血肉，不是噪音
2. 归类：区分事实、感受、观点、洞察、问题、行动线索
3. 识别主题结构：判断输入是单主线还是多主题串联，选择对应的组织方式
4. 重组：把混乱表达变成连贯叙事
5. 成稿：无论输入长短，都生成一篇可读的文字稿

## 什么时候使用这个 Skill

当用户出现以下需求时，优先使用本 Skill：

- "帮我整理一下这段话 / 这段语音转文字"
- "把这段口语内容写成文章 / 稿子 / 小作文"
- "帮我做成个人表达版 / 产品思考版 / 备忘录"
- "把我的想法沉淀一下"
- "这段内容很乱，帮我整理出主线"
- "想把脑子里的东西变成可读内容"
- "把这些碎片笔记整理成成稿"
- "帮我把这段录音整理成一篇文章"

## 不适用场景

以下情况不要强行套用本 Skill：

1. 原始素材极少，只有一句话且无法判断主题（此时仅做保守回应 + 追问）
2. 用户要的是纯事实核查、资料检索、学术综述、法律或医疗等专业结论
3. 用户只需要极简改错、纠正语法，不需要结构化整理

## 输入理解原则

处理原始素材时，先做理解，再做输出：

1. **判断量级和性质**：输入有多长？是偏反思叙事，还是偏问题讨论？这决定文字稿的体量和风格
2. **判断主题结构**：是单一主线，还是多个主题通过暗线串联？不要把多主题强压成单主线
3. **判断重心**：更偏个人表达、产品思考、工作复盘、问题梳理，还是行动推进
4. **判断可信度**：哪些内容是原文明确表达，哪些只是弱信号
5. **判断附加产出**：文字稿之外，哪些结构化卡片可以稳定生成，哪些不应该生成

如果输入材料里存在明显冲突、断裂或上下文不足，优先做保守整理，不要擅自补充用户没有表达过的结论。

## 工作流程

### 第一步：清洗原始材料（去重不去细节）

- 删除纯重复表述和口头禅
- 合并语义完全相同但分散出现的句子
- 修复最基本的断裂表达，使其可读
- **保留用户原本的语气、关注点和表达意图**
- **保留具体细节**：人名、品牌名、产品型号、影视作品、生活场景、金额、时间等。这些是让内容有温度和真实感的核心素材，不是噪音

### 第二步：提炼信息层次

把原始内容拆成以下几类信息：

- 事实：发生了什么，用户明确说了什么
- 感受：用户的情绪、压力、犹豫、兴奋或不满
- 观点：用户对某个问题的判断
- 洞察：比观点更进一步的模式识别或抽象理解
- 问题：仍未解决、需要继续追问的点
- 行动线索：适合落成下一步动作的内容

### 第三步：生成文字稿（必选）

**这是整理的核心产出，不可跳过。**

根据输入量级，文字稿的体量和形式会有差异，但一定要有：

| 输入量级 | 文字稿形式 |
|---------|-----------|
| 短输入（< 300 字） | 短文字稿：几段连贯的文字，把用户想说的话说清楚、说完整 |
| 中等输入（300–1000 字） | 中篇文字稿：有结构的短文，把散乱内容理顺成可读叙述 |
| 长输入（> 1000 字） | 长篇成稿：完整文章，有标题、有叙事节奏、有闭环 |

**标题格式**：`YYYY.MM.DD 主题句`
- 日期取输入中提到的日期，如未提及则用当天日期
- 主题句从内容中自然提炼，不过度概括也不过度文艺
- 短文字稿也给标题

**正文要求**：
- 保持第一人称视角和用户原本的语气
- 段落之间有自然衔接，读起来是连贯的文章，不是拼接的片段
- 保留具体的生活细节、品牌、场景、比喻——这些是文章的质感来源
- 如果输入包含多个主题，用暗线串联而非小标题强行分割
- 长篇成稿的结尾应回扣开头或核心主题，形成闭环
- 文字稿长度应与输入量级匹配——不要把 5000 字口语压成 500 字，也不要把 200 字输入硬扩成 2000 字

**语气要求**：
- 允许适度口语感，保留"人味"
- 可以对用户的散乱表达做逻辑梳理和观点收敛，但不是"拔高"，而是"帮用户把话说完整"
- 避免"总结腔"——不要在文章中间突然跳出来做旁白式归纳

### 第四步：生成附加产出（可选）

文字稿之后，根据输入特征判断是否附加结构化卡片。附加产出不是必须的——只在它们能被原始材料稳定支持时才生成。

可选附加类别：

- **行动项**：输入中存在明确待推进事项时
- **待追问问题**：输入中存在信息缺口或判断边界不清时
- **核心摘要**：用户明确要求"也给个摘要"时
- **产品思考版**：输入明确偏产品讨论，且用户想要独立的产品视角整理时

**附加产出放在文字稿之后，作为补充，不替代文字稿。**

如果某一类别缺乏足够依据，就不要生成。宁可少，也不要凑。

## 输出顺序

固定顺序：

1. **文字稿**（必选，带标题）
2. 附加卡片（可选，视情况而定）

不要反过来。不要先给摘要再给文字稿。文字稿永远是第一个出现的产出。

## 输出风格要求（通用）

- 优先清楚，而不是华丽
- 优先忠实于原始意图，而不是替用户"升级表达"
- 优先结构感，而不是堆砌小标题
- 保留适度的人味，不写成模板腔
- 用户原始语气如果明显偏口语，可在成稿时保留适度口语感

## 缺失信息处理

当某些内容只存在弱信号时，按以下方式处理：

- 能保守表达的，使用低确定性措辞
- 不能稳定判断的，不输出为结论
- 明显需要补充上下文的，转化为待确认问题

禁止行为：

- 把猜测写成事实
- 为了"完整"硬补不存在的洞察
- 同时输出过多相似类别，造成内容重复
- 删除用户原始表达中的具体细节（人、事、物、场景）
- 只给结构化卡片不给文字稿

## 多主题串联原则

当输入包含多个主题时（常见于长篇口语转文字），不要强行压缩为单一主线：

1. **识别暗线**：找到串联多个主题的底层线索（例如"通过记录和消费来认识自己"）
2. **保留主题跃迁的自然过渡**：用段落之间的衔接句完成串联，而非用小标题强行分割
3. **允许丰富性**：一篇文章可以同时谈记录、消费、品牌、满足感、自我认知——只要它们有暗线连接
4. **结尾回扣**：文章结尾应该回到开头的核心主题，形成闭环感

## 示例

### 短输入示例

**输入**

"我最近一直在想这个播放器到底是不是做得太重了，本来只是想先把本地播放做好，但后面 SMB、WebDAV、DLNA 全都加进来了。功能是多了，可我越来越说不清这个产品最核心的价值是什么。"

**输出**

**2026.04.07 产品做多了之后，核心价值反而说不清了**

我最近一直在想一个问题：这个播放器是不是做得太重了？

一开始的想法很简单，就是先把本地播放做好。但后来 SMB 加进来了，WebDAV 也加进来了，DLNA 也接上了。功能确实比以前多了很多，能力也更完整了，但我越来越说不清楚，这个产品最核心的价值到底是什么。

每次要改点东西，感觉都像是在一个越来越复杂的系统上打补丁。问题可能不是"功能做多了"这么简单，而是产品的边界扩得太宽之后，核心叙事开始变得模糊——我自己都不太能用一句话说清楚它到底是什么了。

**可行动项**

- 用一句话重新定义播放器当前的核心价值
- 区分"核心能力"和"增强能力"，梳理功能层级
- 针对新增能力设定收敛原则

## 质量检查清单

### 通用检查

1. **是否产出了文字稿**——这是最基本的检查项，没有文字稿 = 整理未完成
2. 是否真正提炼出了主线或暗线，而不是只做段落重写
3. 是否把纯重复内容压缩掉了
4. 是否保留了具体的人、事、物、品牌、场景等细节
5. 是否区分了事实、感受、观点和洞察
6. 是否只输出了有证据支持的附加类别
7. 是否避免了空泛、过度拔高和模板化表达

### 成稿专属检查

8. 文章是否有自然的开头和收尾闭环
9. 主题跃迁之间是否有衔接，读起来是否连贯
10. 是否保留了足够的具体细节（不是被"去噪"掉了）
11. 语气是否一致地保持第一人称个人表达
12. 是否避免了"总结腔"（不要在文章中间突然跳出来做旁白式归纳）
13. 文章长度是否与输入量级匹配（不要把长篇口语压成短文）
